# 🛡️ Alchemist Defense Strategy: Deep Dive Q&A

본 문서는 **Unity Performance Alchemist** 프로젝트의 실질적 가용성과 안정성에 대한 기술적 의구심을 방어하기 위해 작성되었습니다.

---

## 1. Safety & Stability (안전성)

**Q: AI가 생성한 코드가 논리적 버그(Logic Bug)를 일으키면 어떡하나? 성능은 올라도 게임 루프가 깨질 수 있다.**
- **Defense**: 
    - 본 프로젝트는 **'최적화 가설 검증'**이 핵심입니다. 현재는 FPS/GC 지표에 집중하고 있지만, 실제 상용 환경에서는 **자동화된 단위 테스트(Unit Test)**와 연동되어야 합니다.
    - AI가 리팩토링한 후 `EditMode/PlayMode Test`를 수행하고, 모든 테스트를 통과한 경우에만 최종 채택하는 파이프라인 확장이 가능합니다. (현재 시스템은 크래시 및 컴파일 에러에 대한 1차 롤백 장치 구비)

**Q: AI가 무한 루프 코드를 만들어서 유니티가 뻗어버리면?**
- **Defense**: 
    - `PerformanceAlchemist.cs` 내부의 **Benchmark Watchdog** 로직이 해결책입니다. 
    - 벤치마크 실행 중 일정 시간(15초) 내에 프레임 응답이 없으면 시스템이 강제로 실행을 중단하고 즉시 이전의 안전한 코드로 롤백합니다.

---

## 2. Practicality & Scalability (실무 적용성)

**Q: 대규모 프로젝트는 스크립트 간 의존성이 복잡하다. 단일 파일 리팩토링이 가능한가?**
- **Defense**: 
    - 매우 유효한 지적입니다. Alchemist는 현재 **'Leaf Component(의존성 끝단)'** 최적화에 특화되어 있습니다. 
    - 전역적인 리팩토링을 위해서는 AI에게 프로젝트의 메타데이터와 의존성 그래프를 제공하는 **Context Provider** 기능이 보강되어야 하며, 이는 현재 `Codebase Investigator`와 같은 에이전트 기술로 충분히 확장이 가능한 영역입니다.

**Q: 왜 굳이 무거운 GPT-4 대신 로컬 모델(Llama 3.2)을 고집했나?**
- **Defense**: 
    - **보안(Privacy)**과 **비용(Cost)** 때문입니다. 상용 프로젝트의 소스 코드는 기업의 핵심 자산입니다. 외부 API로 유출되는 것을 원치 않는 기업들을 위해 **'온디바이스 AI 성능 공학'**이라는 비전을 제시했습니다. 
    - 또한, 최적화 루프는 수십 번 반복되는데, 이때 발생하는 API 비용을 0으로 만들어 경제적 타당성을 확보했습니다.

---

## 3. Engineering Philosophy (엔지니어링 철학)

**Q: 뻔한 문제를 푼 것 아닌가? (O(n^2), Instantiate 등)**
- **Defense**: 
    - 뻔한 문제일수록 **'양이 많고 고치기 지루합니다.'** 10년 된 레거시 프로젝트에서 이런 뻔한 병목들을 사람이 일일이 찾는 것은 인적 낭비입니다. 
    - 저는 **'엔지니어링의 지루한 부분(Drudgery)을 자동화'**하여, 인간 엔지니어가 더 창의적인 설계에 집중할 수 있게 만드는 **'시스템 파이프라인 설계'** 역량을 증명하는 데 집중했습니다.

**Q: 앞으로의 고도화 계획은?**
- **Defense**: 
    - 현재의 소스 코드 수준 최적화를 넘어, **IL2CPP 빌드 후의 기계어 레벨 벤치마킹**과 **셰이더(HLSL) 연산 최적화**로 영역을 넓힐 계획입니다. 
    - 특히 유니티의 **Addressables 시스템의 메모리 릭**을 자율적으로 추적하고 수정하는 미션을 다음 목표로 삼고 있습니다.

---
**이 도구는 AI 코딩 비서가 아닙니다. AI를 부품으로 사용하는 '지능형 성능 제어 시스템'입니다.**
